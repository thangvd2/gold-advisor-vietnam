# Plan 08-03: Integration into Composite Scorer + Pipeline + Dashboard Display

## Objective

Integrate seasonal demand modifier and State Bank policy override into the signal pipeline, composite scorer, reasoning generator, and dashboard display. This wires everything together for the user to see.

## Context

- `src/engine/composite.py` computes composite signal from weighted factors
- `src/engine/pipeline.py` assembles factors and calls composite scorer
- `src/engine/reasoning.py` generates one-line reasoning text
- `src/engine/modes.py` has mode weights (gap, spread, trend, fx_trend, gold_trend)
- `src/engine/seasonal.py` (from 08-01) provides seasonal modifier
- `src/engine/policy.py` (from 08-02) provides policy override data
- `templates/partials/signal_card.html` displays signal on dashboard
- Per PITFALLS: "State Bank actions override all other signals" and "seasonal patterns EXPLAIN gap widening"

## Tasks

### Task 1: Composite Scorer Override Integration (TDD)

**type:** auto
**tdd:** true

**behavior:** Update `compute_composite_signal()` to accept optional `policy_override` and `seasonal_modifier` parameters:
- If `policy_override["has_override"]` is True, cap confidence at `policy_override["confidence_cap"] * 100`
- Apply `seasonal_modifier` to reduce final confidence (confidence * seasonal_modifier)
- Policy override takes priority over seasonal modifier

**implementation:** Modify `src/engine/composite.py` to apply confidence caps. Seasonal modifier is multiplicative. Policy override is a hard cap.

**done:** Signal confidence is capped by policy events and reduced by seasonal demand.

### Task 2: Pipeline Integration (TDD)

**type:** auto
**tdd:** true

**behavior:** Update `compute_signal()` in pipeline to:
1. Compute seasonal signal from current month
2. Compute policy override from database
3. Pass seasonal_modifier and policy_override to composite scorer
4. Pass seasonal and policy context to reasoning generator

**implementation:** Import seasonal and policy modules. Compute seasonal modifier from `datetime.now().month`. Compute policy override via `compute_policy_signal()`. Pass both to `compute_composite_signal()` and `generate_reasoning()`.

**done:** Pipeline produces signals with seasonal and policy adjustments applied.

### Task 3: Reasoning Generator Update (TDD)

**type:** auto
**tdd:** true

**behavior:** Update `generate_reasoning()` to append seasonal and policy context:
- Seasonal: "Seasonal demand is [low/medium/high] for [month name] — [explanation]"
- Policy: "State Bank policy alert: [event summary]" (when active high-severity events exist)
- When both are present: append both clauses

**implementation:** Add seasonal_context and policy_context parameters. Build context strings and append to reasoning body.

**done:** Reasoning includes seasonal demand level and active policy event summaries.

### Task 4: Dashboard Signal Card Update (TDD)

**type:** auto
**tdd:** true

**behavior:** Update signal card partial to display:
- Seasonal demand badge (e.g., "High Demand Season") when month is high/very_high
- Policy alert banner when active policy events exist (e.g., "State Bank Policy Active")
- Both shown as contextual indicators below the reasoning text

**implementation:** Pass `seasonal_info` and `policy_info` to template context. Add conditional HTML blocks for seasonal badge and policy alert banner.

**done:** Dashboard shows seasonal and policy context on signal card.

### Task 5: API Response Update

**type:** auto

Update the dashboard signal API and signal partial endpoints to pass seasonal and policy data to the template. Update `GET /dashboard/signal` JSON response to include `seasonal_demand_level` and `active_policy_events` fields.

**done:** API responses include seasonal and policy context data.
