# Plan 08-01: Seasonal Demand Model

## Objective

Build a Vietnamese seasonal demand calendar that maps months to demand levels (low/medium/high/very_high) and computes a seasonal confidence modifier for the signal engine. Seasonal patterns EXPLAIN gap widening — they don't generate buy/sell signals themselves.

## Context

- `src/engine/types.py` defines `SignalFactor` with `name`, `direction`, `weight`, `confidence`
- Per PITFALLS.md: "A VND 5M gap during Tet is normal (structural demand). A VND 5M gap during quiet August is unusual."
- Seasonal factor: doesn't change direction, only adjusts confidence (reduce during high-demand periods)
- Lunar calendar events: Tet (Jan-Feb), Vu Lan (Aug lunar 7th), wedding season (Mar-Apr, Nov-Dec)

## Tasks

### Task 1: Vietnamese Seasonal Calendar (TDD)

**type:** auto
**tdd:** true

**behavior:** `get_seasonal_demand_level(month: int) -> str` returns demand level for a given month (1-12). Mapping:
- January: "very_high" (pre-Tet accumulation)
- February: "very_high" (Tet buying)
- March: "medium" (post-Tet selling, wedding season start)
- April: "medium" (wedding season)
- May: "low" (summer doldrums)
- June: "low" (summer doldrums)
- July: "low" (summer doldrums)
- August: "medium" (Vu Lan)
- September: "low" (quiet period)
- October: "low" (quiet period)
- November: "high" (pre-Tet accumulation, wedding season)
- December: "high" (pre-Tet accumulation, wedding season)

Invalid months raise ValueError.

**implementation:** Simple lookup dict in `src/engine/seasonal.py`.

**done:** Function returns correct demand level for all 12 months.

### Task 2: Seasonal Confidence Modifier (TDD)

**type:** auto
**tdd:** true

**behavior:** `compute_seasonal_modifier(month: int) -> float` returns a confidence multiplier based on seasonal demand:
- "very_high" → 0.7 (reduce confidence significantly — gap widening expected)
- "high" → 0.85 (moderate reduction)
- "medium" → 1.0 (no adjustment)
- "low" → 1.0 (no adjustment — gaps during quiet periods are more meaningful)

The modifier is < 1 during high-demand periods because gap widening is expected and not anomalous.

**implementation:** Call `get_seasonal_demand_level()`, map to multiplier. Returns float between 0.7 and 1.0.

**done:** Returns correct modifier for each demand level.

### Task 3: Seasonal Signal Factor (TDD)

**type:** auto
**tdd:** true

**behavior:** `compute_seasonal_signal(month: int, weight: float = 0.0) -> SignalFactor` returns a SignalFactor:
- `name`: "seasonal"
- `direction`: 0.0 (seasonal doesn't drive direction)
- `weight`: 0.0 (seasonal doesn't contribute to score direction)
- `confidence`: stores the seasonal modifier as a metadata value in a custom field

The seasonal factor does NOT influence the buy/sell direction. It only provides data for the reasoning module and composite scorer to use as a confidence adjustment.

**implementation:** Create SignalFactor with direction=0.0, weight=0.0, confidence=modifier value. The composite scorer will use this factor differently from others (see Plan 08-03).

**done:** Returns SignalFactor with correct seasonal modifier in confidence field.

## Verification

1. `get_seasonal_demand_level()` returns correct levels for all 12 months
2. `compute_seasonal_modifier()` returns correct multipliers
3. `compute_seasonal_signal()` returns SignalFactor with modifier in confidence
4. Seasonal factor has direction=0.0 and weight=0.0 (no directional influence)
5. All tests pass
